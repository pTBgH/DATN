\chapter[CƠ CHẾ HOẠT ĐỘNG CỦA CÁC DỊCH VỤ]{Cơ chế hoạt động của các dịch vụ}
\label{ch:appendix}

Phụ lục này mô tả cơ chế hoạt động, cấu hình nguồn minh họa và kết quả vận hành của các dịch vụ nghiệp vụ cũng như các giải pháp bảo mật trong hệ thống Job7189. Nội dung trong phụ lục tập trung vào cách vận hành và đối chiếu cấu hình, bổ sung cho phần thiết kế và kiểm chứng ở các chương chính.

Hệ thống được tổ chức thành các phân vùng ở cấp độ không gian tên (namespace) để tạo ranh giới vận hành rõ ràng. Dưới đây là cơ chế hoạt động, cấu hình minh họa và kết quả vận hành của các thành phần chính.

\section{Giao diện frontend và Cloudflare Pages}
Job7189 có hai frontend Next.js: \texttt{atd\_frontend} dành cho ứng viên và \texttt{rct\_frontend} dành cho nhà tuyển dụng, đồng thời phục vụ một số thao tác quản trị. Hai frontend này không được triển khai như tác vụ trong cụm Kubernetes; mã nguồn có cấu hình \texttt{wrangler.toml} để đóng gói bằng \texttt{@cloudflare/next-on-pages} và đưa lên Cloudflare Pages.

Về luồng truy cập, trình duyệt tải giao diện từ Cloudflare Pages. Khi người dùng đăng nhập, frontend gọi API của Keycloak để lấy JWT, sau đó gửi yêu cầu nghiệp vụ tới Kong Gateway. Vì vậy, cụm Kubernetes vẫn là nơi xử lý backend, định danh, dữ liệu và chính sách bảo mật; Cloudflare chủ yếu đảm nhận phần phân phối giao diện.

\begin{minted}[fontsize=\scriptsize,breaklines]{text}
atd_frontend  -> job7189-atd  -> Cloudflare Pages
rct_frontend  -> job7189-rct  -> Cloudflare Pages
\end{minted}

\section{Cổng giao tiếp Kong}
Cổng giao tiếp Kong đóng vai trò là điểm kiểm soát chính cho các yêu cầu API đi vào backend từ mạng ngoài.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Cổng giao tiếp vận hành ở chế độ khai báo không dùng cơ sở dữ liệu (DB-less mode) thông qua tệp cấu hình nguồn \texttt{kong.yml}. Với các route yêu cầu xác thực, trình cắm \texttt{jwt} của Kong kiểm tra chữ ký và thời hạn của JWT do Keycloak phát hành. Sau đó, trình cắm tiền chức năng (pre-function plugin) viết bằng Lua giải mã phần payload đã được xác minh, gửi ngữ cảnh yêu cầu đến OPA để phân biệt route công khai và route cần phiên đăng nhập. Nếu OPA cho phép, Kong chuyển tiếp yêu cầu đến microservice đích và gắn lại tiêu đề \texttt{Authorization} để Laravel tiếp tục thực hiện phân quyền nghiệp vụ theo workspace.
    \item \textbf{Cấu hình định tuyến và trình cắm Lua}: Đoạn cấu hình khai báo dịch vụ nghiệp vụ và trình cắm tiền chức năng được mô tả như sau:
\begin{minted}[fontsize=\scriptsize,breaklines]{yaml}
# Trích từ cấu hình nguồn infras/kong/kong.yml
services:
  - name: identity-service
    url: http://identity-service.job7189-apps.svc.cluster.local:80
    routes:
      - name: identity-profiles
        paths: ["/api/recruiters/profile"]
        strip_path: false
        plugins:
          - name: jwt
            config:
              claims_to_verify: [exp]

plugins:
  - name: pre-function
    config:
      access:
        - |
          local cjson   = require("cjson.safe")
          local http    = require("resty.http")
          if kong.request.get_method() == "OPTIONS" then return end
          
          local OPA_URL     = "https://opa.security.svc.cluster.local:8181/v1/data/zta/authz/allow"
          local FAIL_CLOSED = true
          local req_method  = kong.request.get_method()
          local req_path    = ngx.var.uri or "/"
          local auth        = kong.request.get_header("authorization") or ""
          local token       = auth:match("^[Bb]earer%s+(.+)$") or ""
          
          local function b64url_decode(s)
            s = s:gsub("-", "+"):gsub("_", "/")
            local pad = 4 - (#s % 4)
            if pad < 4 then s = s .. string.rep("=", pad) end
            return ngx.decode_base64(s)
          end
          
          local claims = {}
          if token ~= "" then
            local _, payload_b64, _ = token:match("([^.]+)%.([^.]+)%.([^.]+)")
            if payload_b64 then
              local raw = b64url_decode(payload_b64)
              if raw then claims = cjson.decode(raw) or {} end
            end
          end
          
          local segments = {}
          for seg in req_path:gmatch("[^/]+") do table.insert(segments, seg) end
          
          local opa_input = {
            input = {
              method = req_method,
              path = req_path,
              path_segments = segments,
              jwt = claims,
            }
          }
          
          local httpc = http.new()
          httpc:set_timeout(1000)
          local res, err = httpc:request_uri(OPA_URL, {
            method = "POST",
            body = cjson.encode(opa_input),
            headers = { ["Content-Type"] = "application/json" },
            ssl_verify = false,
          })
          
          if not res or res.status ~= 200 then
            if FAIL_CLOSED then
              return kong.response.exit(503, {
                message = "authorization service unavailable"
              })
            end
            return
          end
          local decoded = cjson.decode(res.body or "") or {}
          if decoded.result ~= true then
            return kong.response.exit(403, {
              message = "forbidden",
              reason = "OPA denied user-authz",
              user = claims.preferred_username or "anonymous",
              azp = claims.azp or ""
            })
          end
          
          if token ~= "" then
            ngx.req.set_header("authorization", "Bearer " .. token)
          end
\end{minted}
    \textit{Ghi chú: Trong đoạn cấu hình minh họa, tùy chọn \texttt{ssl\_verify = false} được dùng để đơn giản hóa kết nối nội bộ trong môi trường phòng thí nghiệm. Khi vận hành thật, kết nối từ Kong tới OPA cần xác minh chứng chỉ hoặc cơ chế xác thực tương đương để bảo đảm đúng đầu cuối ở tầng TLS.}
    \item \textbf{Kết quả chạy lệnh minh họa}: Khi gửi yêu cầu không có mã thông báo hoặc khi mã thông báo bị OPA từ chối:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
# Trường hợp không gửi mã thông báo JWT (Kong chặn trực tiếp tại route):
$ curl -i -H "Host: api.job7189.local" \
  http://100.108.231.127:30000/api/recruiters/profile
HTTP/1.1 401 Unauthorized
Content-Type: application/json
Connection: keep-alive
Server: kong/3.6

{"message":"Unauthorized"}

# Trường hợp gửi mã thông báo hợp lệ nhưng bị chính sách OPA từ chối truy cập:
$ curl -i -H "Host: api.job7189.local" \
  -H "Authorization: Bearer <valid_jwt>" \
  http://100.108.231.127:30000/api/admin/users
HTTP/1.1 403 Forbidden
Content-Type: application/json
Connection: keep-alive
Server: kong/3.6

{"message":"forbidden","reason":"OPA denied user-authz","user":"recruiter1"}
\end{minted}
\end{itemize}

\section{Dịch vụ xác thực Keycloak}
Keycloak đóng vai trò là trung tâm định danh người dùng cuối, chịu trách nhiệm quản lý thông tin tài khoản của nhà tuyển dụng và ứng viên.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Keycloak thực hiện xác thực người dùng dựa trên realm \texttt{job7189}. Sau khi xác thực thành công, Keycloak ký số mã thông báo truy cập (JWT) bằng khóa riêng tư. Khóa công khai dùng để xác minh chữ ký được cung cấp qua điểm cuối JWKS, cho phép Kong và các microservices đối chiếu cục bộ. Theo mô hình hiện tại trong cơ sở tri thức, Keycloak không mang các vai trò nghiệp vụ như \texttt{admin}, \texttt{recruiter} hay \texttt{member}; frontend dùng client \texttt{recruiter-app-dev} hoặc \texttt{candidate-app-dev}, còn Laravel đọc claim \texttt{azp} và dữ liệu \texttt{workspace\_members} để quyết định quyền nghiệp vụ.

\section{Hệ thống định danh ứng dụng SPIRE}
SPIRE cung cấp cơ chế định danh tự động và bảo mật cho các tác vụ trong container (workload) thông qua chứng thư định danh SVID dưới dạng X.509 hoặc JWT.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Tác nhân SPIRE (SPIRE Agent) hoạt động trên từng nút và lắng nghe trên một socket Unix cục bộ. Khi một Pod khởi chạy, SPIRE Agent truy vấn thông tin định danh của Pod từ hệ thống Kubernetes (bao gồm không gian tên, tài khoản dịch vụ và các nhãn của Pod) để thực hiện xác thực thực thể, sau đó cấp phát chứng thư định danh SVID tương ứng cho Pod thông qua Workload API. Giao diện API này được liên kết trực tiếp vào Pod nhờ trình điều khiển CSI (\texttt{csi.spiffe.io}).
    \item \textbf{Cấu hình ClusterSPIFFEID}: Đoạn mã dưới đây mô tả chính sách đăng ký định danh tự động cho các dịch vụ nghiệp vụ:
\begin{minted}[fontsize=\scriptsize,breaklines]{yaml}
# Trích từ cấu hình nguồn infras/k8s-yaml/spire/cluster-spiffe-ids.yaml
apiVersion: spire.spiffe.io/v1alpha1
kind: ClusterSPIFFEID
metadata:
  name: zta-default-workload-identity
spec:
  spiffeIDTemplate: "spiffe://zta.job7189/ns/{{ .PodMeta.Namespace }}/sa/{{ .PodSpec.ServiceAccountName }}"
  podSelector:
    matchExpressions:
      - key: cilium.zta/role
        operator: Exists
  namespaceSelector:
    matchExpressions:
      - key: kubernetes.io/metadata.name
        operator: In
        values:
          - job7189-apps
          - gateway
          - vault
          - security
          - monitoring
          - data
          - management
  dnsNameTemplates:
    - "{{ index .PodMeta.Labels \"app\" }}.{{ .PodMeta.Namespace }}.svc.cluster.local"
\end{minted}
    \item \textbf{Kết quả chạy lệnh minh họa}: Kiểm tra danh sách các thực thể microservices được đăng ký trên Máy chủ SPIRE (SPIRE Server):
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}

  -- /opt/spire/bin/spire-server entry show

Selectors     : k8s:sa:identity-service
Selectors     : k8s:pod-label:app:identity-service
\end{minted}
\end{itemize}

\section{Hệ thống quản lý thông tin mật Vault}
Vault quản lý các thông tin cấu hình nhạy cảm và cấp phát tài khoản cơ sở dữ liệu động với thời hạn ngắn.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Thay vì sử dụng phương thức xác thực Kubernetes thông thường, hệ thống được cấu hình phương thức xác thực JWT (\texttt{auth/jwt}). Khi microservice nghiệp vụ như \texttt{identity-service} khởi chạy, tiến trình phụ \texttt{spiffe-helper} trong Pod lấy JWT-SVID với audience \texttt{vault} từ SPIRE Agent qua Workload API và ghi vào tệp \texttt{/spire-jwt/jwt\_svid.token} trên bộ nhớ tạm. \texttt{vault-agent} dùng JWT-SVID này để đăng nhập Vault qua \texttt{auth/jwt/login}; Vault đối chiếu issuer, audience và subject SPIFFE, ví dụ \texttt{spiffe://zta.job7189/ns/job7189-apps/sa/identity-service}, rồi cấp token với policy tương ứng. Sau khi xác thực thành công, Vault Agent tự động lấy và lưu trữ tài khoản MySQL động có thời hạn ngắn. Trong môi trường mô phỏng, thời hạn này được đặt thấp, khoảng 10--15 phút, để dễ quan sát việc xoay vòng bí mật; giá trị trong kết quả lệnh chỉ thể hiện cấu hình tại thời điểm minh họa.
    \item \textbf{Cấu hình chính sách Vault}: Đoạn mã cấu hình quyền hạn cho phép microservice nghiệp vụ đọc thông tin tài khoản động:
\begin{minted}[fontsize=\scriptsize,breaklines]{text}
# Vault Policy: identity-service-policy
path "secret/data/laravel-common" {
  capabilities = ["read"]
}
path "database/creds/identity-service" {
  capabilities = ["read"]
}
path "secret/data/identity-service" {
  capabilities = ["read"]
}
\end{minted}
    \item \textbf{Kết quả chạy lệnh minh họa}: Mô phỏng câu lệnh đăng nhập và đọc thông tin tài khoản động MySQL được thực hiện tự động:

Cơ chế này làm giảm khả năng pod khác tự nhận vai trò của \texttt{identity-service}; tuy nhiên hiệu quả phụ thuộc vào việc ràng buộc đúng trust domain, audience, policy Vault và quyền truy cập tới Workload API. Trong ví dụ này, thời hạn hiệu lực của token Vault dài hơn thời hạn của tài khoản cơ sở dữ liệu động; Vault Agent cần gia hạn token đúng hạn để tránh gián đoạn khi lấy bí mật mới.}

\section{Bộ quyết định chính sách PDP}
Bộ quyết định chính sách PDP thực hiện tính toán điểm tin cậy và cập nhật nhãn phân hạng mức độ tin cậy của các Pod dựa trên kết quả quét lỗ hổng thực tế.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Bộ điều khiển PDP định kỳ quét nhãn Pod và các báo cáo lỗ hổng thu được từ Trivy Operator để tính toán mức độ an toàn của Pod, từ đó gán nhãn phân hạng tin cậy (\texttt{high}, \texttt{medium}, \texttt{low}) trực tiếp lên Pod tương ứng. Công thức thực nghiệm dùng các trọng số cấu hình qua biến môi trường: thiếu nhãn tối đa trừ 30 điểm, CVE Critical trừ 50 điểm, CVE High trừ 20 điểm, cộng thêm hệ số nhạy cảm theo tier khi Pod đã có vấn đề về posture. Trục CISA KEV cũng được hỗ trợ nhưng mặc định có trọng số bằng 0 cho đến khi ConfigMap danh mục KEV được nạp và bật trọng số tương ứng. Nhãn \texttt{zta.job7189/score-bucket} được Cilium sử dụng làm thuộc tính định danh để áp dụng các chính sách phân vùng mạng.
    \item \textbf{Kết quả gán nhãn động}: Xem danh sách các nhãn tin cậy được PDP gán động trên các Pod ứng dụng trong cụm:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
$ kubectl get pods -n job7189-apps \

\begin{itemize}

Lớp chính sách mạng của Cilium hoạt động ở lớp 3/4 (địa chỉ IP và cổng), đồng thời có thể kiểm soát lớp 7 (đường dẫn HTTP và phương thức) thông qua dịch vụ proxy Envoy tích hợp sẵn.     \item \textbf{Cấu hình minh họa (Cilium Network Policy L7)}: Đoạn cấu hình chính sách mạng cho phép gọi API nội bộ có giới hạn đường dẫn từ \texttt{identity-service} sang \texttt{workspace-service} ở tầng ứng dụng L7:

\section{Các microservices nghiệp vụ}
Các microservices nghiệp vụ được cấu hình chạy song song với các container phụ trợ trong cùng một Pod để hỗ trợ cập nhật thông tin bảo mật tự động.
\begin{itemize}
    \item \textbf{Cơ chế hoạt động}: Khi \texttt{vault-agent} lấy và ghi tài khoản MySQL mới vào tệp \texttt{.env} trên phân vùng bộ nhớ tạm (tmpfs), tiến trình phụ \texttt{env-watcher} sẽ phát hiện sự thay đổi nội dung tệp tin bằng mã băm MD5 và gửi yêu cầu HTTP POST nội bộ tới container \texttt{app} để cập nhật cấu hình hệ thống tức thì. Nếu yêu cầu HTTP thất bại, tiến trình sẽ gửi tín hiệu \texttt{SIGUSR2} tới tiến trình \texttt{php-fpm} master trong Pod để khởi động lại máy chủ ứng dụng một cách an toàn.
    \item \textbf{Đoạn mã giám sát cấu hình}:
\begin{minted}[fontsize=\scriptsize,breaklines]{sh}
# Trích từ cấu hình k8s-management/charts/laravel-app/files/env-watcher.sh
#!/bin/sh
set -eu
APP_DIR=${APP_DIR:-/app-secrets}
RELOAD_URL=${RELOAD_URL:-http://127.0.0.1:8000/api/internal/reload-db}
RELOAD_TOKEN=${RELOAD_TOKEN:-}
CHECK_INTERVAL=${CHECK_INTERVAL:-5}
MAX_NOTIFY_RETRIES=${MAX_NOTIFY_RETRIES:-3}

md5_of() { md5sum "$1" 2>/dev/null | cut -d' ' -f1 || echo ""; }

notify_app() {
  if [ -n "$RELOAD_TOKEN" ]; then
    i=0
    while [ "$i" -lt "$MAX_NOTIFY_RETRIES" ]; do
      if wget -q -O /dev/null --post-data="" --header="X-Internal-Token: $RELOAD_TOKEN" -T 3 "$RELOAD_URL"; then
        echo "$(date -Is) notify: HTTP reload success"
        return 0
      fi
      i=$((i+1))
      sleep 1
    done
  fi

  for p in /proc/[0-9]*/cmdline; do
    if grep -ql 'php-fpm: master' "$p" 2>/dev/null; then
      pid=$(echo "$p" | cut -d'/' -f3)
      if kill -USR2 "$pid" 2>/dev/null; then
        echo "$(date -Is) notify: sent USR2 to php-fpm master (pid $pid)"
        return 0
      fi
    fi
  done
  return 1
}

LAST=""
if [ -f "$APP_DIR/.env" ]; then LAST=$(md5_of "$APP_DIR/.env") fi

while true; do
  if [ -f "$APP_DIR/.env" ]; then
    SUM=$(md5_of "$APP_DIR/.env")
    if [ "$SUM" != "$LAST" ]; then
      LAST="$SUM"
      echo "$(date -Is) env-watcher: .env changed, notifying app"
      if ! notify_app; then
        echo "$(date -Is) env-watcher: notify failed (HTTP + signal)"
      fi
    fi
  fi
  sleep "$CHECK_INTERVAL"
done
\end{minted}
    \textit{Ghi chú: Mã MD5 trong tiến trình \texttt{env-watcher} chỉ dùng để phát hiện tệp \texttt{.env} đã thay đổi, không dùng để bảo vệ bí mật hay kiểm chứng tính toàn vẹn trước kẻ tấn công. Vì vậy, điểm yếu va chạm của MD5 không phải cơ chế bảo mật chính trong thiết kế này; nếu cần bảo đảm chống giả mạo, phải dùng chữ ký hoặc HMAC với khóa bí mật.}
    
    \textit{Ghi chú: Endpoint \texttt{/api/internal/reload-db} chỉ phục vụ tiến trình phụ trong cùng pod và được bảo vệ bằng \texttt{RELOAD\_TOKEN}. Token này không phải ranh giới bảo mật mạnh nếu container ứng dụng đã bị chiếm quyền, vì kẻ tấn công trong container có thể lạm dụng endpoint để gây reload lặp lại. Trong triển khai chặt chẽ hơn, endpoint nên chỉ lắng nghe trên localhost, giới hạn tần suất gọi và tách quyền giữa container ứng dụng với container phụ trách reload.}
    \item \textbf{Kết quả chạy lệnh minh họa}: Log hoạt động của Pod ghi nhận tiến trình phụ phát hiện thay đổi bí mật và yêu cầu ứng dụng nạp lại cấu hình:
\begin{minted}[fontsize=\scriptsize,breaklines]{text}
2026-06-13T15:55:02+00:00 env-watcher: .env changed, notifying app
2026-06-13T15:55:02+00:00 notify: HTTP reload success
[2026-06-13 15:55:02] production.INFO: internal.reload-db success

# Nếu endpoint HTTP không phản hồi, env-watcher dùng cơ chế dự phòng:
2026-06-13T16:10:14+00:00 env-watcher: .env changed, notifying app
2026-06-13T16:10:18+00:00 notify: sent USR2 to php-fpm master (pid 43847)
\end{minted}
\end{itemize}

\chapter{HƯỚNG DẪN VẬN HÀNH HỆ THỐNG}
\label{app:operation-guide}

Phần này hướng dẫn quản trị viên thực hiện các thao tác vận hành cơ bản để khởi tạo, kiểm thử và duy trì hệ thống an ninh Zero Trust. Đây là hướng dẫn tóm tắt theo môi trường của đồ án, không phải tài liệu tái tạo đầy đủ cho mọi cụm Kubernetes.

\section{Khởi tạo và cấu hình hệ thống ban đầu}
Kiến trúc an ninh Zero Trust này được thiết kế để hoạt động trên các cụm Kubernetes tiêu chuẩn, nhưng các lệnh dưới đây phản ánh cấu trúc cụm thử nghiệm của đồ án. Điều kiện tiên quyết về hạ tầng là các nút (bao gồm nút điều khiển - Control Plane và các nút làm việc - Worker Node) phải đảm bảo kết nối thông suốt ở lớp mạng (Lớp 3) thông qua mạng LAN vật lý, mạng riêng ảo (như Tailscale) hoặc các giải pháp tương đương, đồng thời hỗ trợ cơ chế định tuyến mạng Pod overlay (như giao thức VXLAN của Cilium CNI). Ngoài ra, cụm cần có quyền truy cập đến kho lưu trữ ảnh vùng chứa (Docker Registry) công cộng hoặc máy chủ Registry nội bộ (ví dụ: địa chỉ thực nghiệm \texttt{100.74.189.43:5443}).

Quá trình triển khai cụm phân tán và thiết lập hạ tầng bảo mật ban đầu được thực hiện tuần tự như sau:

\section{Khởi tạo các nút và liên kết cụm Kubernetes}
Trong môi trường thực tế, cụm Kubernetes được xây dựng trên các máy ảo độc lập kết nối qua mạng riêng. Các tham số \texttt{--server=01}, \texttt{--server=02}, \texttt{--server=03} và \texttt{--server=05} là mã nút trong cụm thử nghiệm: \texttt{srv01} là nút điều khiển, còn \texttt{srv02}, \texttt{srv03} và \texttt{srv05} là các nút làm việc. Quá trình thiết lập cụm được thực hiện bằng cách chạy tuần tự lệnh khởi tạo trên từng nút từ thư mục gốc của dự án:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
# Bước 1: Khởi tạo nút điều khiển chính srv01 (Control Plane)
# Lệnh thực hiện chuẩn bị máy chủ, khởi tạo cụm kubeadm và cài đặt Cilium CNI
$ sudo -E bash knowledge-base/migration/scripts/bootstrap.sh \
  --server=01 --yes

# Bước 2: Gia nhập cụm đối với các nút công việc (Worker Nodes)
# Chạy lệnh join tương ứng trên từng nút làm việc (ví dụ srv02, srv03, srv05):
# Trên srv02:
$ sudo -E bash knowledge-base/migration/scripts/bootstrap.sh \
  --server=02 --yes

# Trên srv03:
$ sudo -E bash knowledge-base/migration/scripts/bootstrap.sh \
  --server=03 --yes

# Trên srv05 (nút chạy Ubuntu 24.04 LTS dành riêng cho cơ sở dữ liệu):
$ sudo -E bash knowledge-base/migration/scripts/bootstrap.sh \
  --server=05 --yes
\end{minted}

\section{Triển khai dịch vụ nghiệp vụ và áp dụng chính sách bảo mật}
Khi các nút trong cụm Kubernetes đã ở trạng thái sẵn sàng, quá trình triển khai dịch vụ nghiệp vụ và kích hoạt các chính sách an ninh Zero Trust sẽ được thực hiện đồng bộ từ thư mục gốc:
\begin{enumerate}
    \item Triển khai các dịch vụ nghiệp vụ và gán nhãn bắt buộc:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
# Triển khai 7 microservices nghiệp vụ vào không gian tên job7189-apps
$ bash 03-deploy-microservices.sh

# Khởi tạo cơ sở dữ liệu ban đầu cho các dịch vụ
$ bash 05-seed-databases.sh

# Gán nhãn Zero Trust bắt buộc lên các Pod vừa tạo để áp dụng các chính sách
$ bash scripts/zta-apply-workload-labels.sh --apply
\end{minted}
    \item Kích hoạt các chính sách bảo mật mạng vi phân vùng và giám sát hệ thống:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
# Áp dụng các chính sách chặn mặc định và cho phép có chọn lọc của Cilium
$ bash infras/k8s-yaml/cilium-policies/apply-zta-microsegmentation.sh

# Triển khai Tetragon để kiểm soát hành vi tiến trình thời gian thực
$ bash 10-deploy-tetragon.sh
\end{minted}
\end{enumerate}

\section{Cập nhật mã nguồn và bổ sung dịch vụ nghiệp vụ}

Phần này mô tả quy trình vận hành khi cần thay đổi mã nguồn một dịch vụ đang có hoặc thêm một dịch vụ Laravel mới vào hệ thống. Về nguyên tắc, thay đổi mã nguồn không chỉ dừng ở việc dựng lại image; image mới còn phải đi qua registry, chữ ký số, Helm values, lớp định danh, bí mật động, chính sách mạng và các bước kiểm tra sau triển khai.

\section{Cập nhật mã nguồn của một dịch vụ đang có}
Với các dịch vụ đã tồn tại như \texttt{identity-service}, \texttt{job-service} hoặc \texttt{communication-service}, quy trình cập nhật được thực hiện theo các bước sau:
\begin{enumerate}
    \item Sửa mã nguồn trong thư mục dịch vụ tương ứng, ví dụ \texttt{src/identity\_service/laravel\_back}. Nếu thay đổi có liên quan đến API, cần kiểm tra lại route, request, response và quyền truy cập tương ứng ở Kong hoặc OPA.
    \item Dựng image mới, đẩy lên registry nội bộ và ký image bằng Cosign. Trong môi trường hiện tại, script \texttt{scripts/build\_and\_deploy.sh} thực hiện các bước này, đồng thời lấy digest của image và ghi lại vào \texttt{image.fullImage} trong tệp values của dịch vụ:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
$ bash scripts/build_and_deploy.sh identity-service v2.8.22
\end{minted}
    \item Helm cập nhật Deployment bằng image mới. Nếu tên release và nhãn \texttt{app} không đổi, các chính sách Cilium, Vault role, SPIFFE ID và luật quan sát hiện có vẫn tiếp tục áp dụng cho pod mới.
    \item Khi pod mới khởi động, \texttt{spiffe-helper} lấy JWT-SVID từ SPIRE, Vault Agent dùng JWT-SVID để đăng nhập Vault, sau đó lấy tài khoản cơ sở dữ liệu động tại đường dẫn theo mẫu \texttt{database/creds/<service-name>}. Container phụ \texttt{env-loader} ghép các tệp môi trường, còn \texttt{env-watcher} gọi endpoint nội bộ để ứng dụng nạp lại thông tin kết nối khi bí mật đổi.
    \item Sau khi rollout xong, cần kiểm tra trạng thái pod, endpoint, nhãn tin cậy và kết nối API:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
$ kubectl rollout status deployment/identity-service -n job7189-apps
$ kubectl get pod -n job7189-apps -l app=identity-service --show-labels
$ python3 test_api.py
\end{minted}
\end{enumerate}

Cơ chế đi qua của một lần cập nhật có thể tóm tắt như sau: mã nguồn được đóng gói thành image; image được đẩy lên registry và ký; Helm đưa digest mới vào Deployment; Kubernetes tạo pod mới; SPIRE và Vault cấp định danh cùng bí mật ngắn hạn; Cilium, Gatekeeper, Tetragon, Trivy và PDP tiếp tục áp chính sách dựa trên image, nhãn và tín hiệu quan sát của pod.

\section{Bổ sung một dịch vụ backend mới}
Khi thêm một dịch vụ mới, cần chuẩn bị đồng thời cả phần ứng dụng và phần chính sách. Nếu chỉ thêm Deployment mà thiếu định danh, bí mật hoặc chính sách mạng, dịch vụ có thể khởi động nhưng không đi qua đầy đủ cơ chế Zero Trust của hệ thống.
\begin{enumerate}
    \item Tạo mã nguồn dịch vụ trong \texttt{src/ten\_service/laravel\_back}, bảo đảm có \texttt{Dockerfile.production}, cấu hình Laravel, route kiểm tra sức khỏe và endpoint nội bộ cần thiết nếu dịch vụ dùng cơ chế nạp lại bí mật động.
    \item Tạo tệp \texttt{k8s-management/values/ten-values.yaml}. Tệp này khai báo \texttt{image.fullImage}, biến môi trường, CSDL tương ứng, Redis nếu cần, host ingress nếu có, cùng các nhãn \texttt{zta.job7189/*} như vai trò, tầng dịch vụ, mức lộ diện và nhóm phụ trách.
    \item Khai báo CSDL hoặc quyền truy cập dữ liệu cho dịch vụ mới. Nếu dịch vụ dùng MySQL động qua Vault, cần có database role tương ứng với tên release, ví dụ \texttt{database/roles/ten-service}, và chính sách cho phép đọc \texttt{database/creds/ten-service}.
    \item Bổ sung định danh workload cho SPIRE theo namespace, ServiceAccount và nhãn pod của dịch vụ. Tên ServiceAccount do chart tạo theo tên release, nên cần giữ thống nhất giữa release Helm, Vault role và SPIFFE ID.
    \item Bổ sung dịch vụ vào quy trình triển khai. Với chart dùng chung \texttt{laravel-app}, mỗi dịch vụ là một release Helm riêng; nếu muốn script tổng \texttt{03-deploy-microservices.sh} quản lý dịch vụ mới, cần thêm tên dịch vụ vào danh sách dịch vụ, danh sách đường dẫn bí mật Vault và cấu hình Helmfile tương ứng.
    \item Khai báo đường đi qua gateway nếu dịch vụ có API cho frontend hoặc người dùng bên ngoài. Khi đó cần thêm service/route trong \texttt{infras/kong/kong.yml}, cập nhật policy OPA nếu route cần phân quyền, và kiểm tra lại client hoặc role trong Keycloak nếu có quyền mới.
    \item Khai báo chính sách mạng Cilium theo nguyên tắc chỉ mở luồng cần thiết: ai được gọi dịch vụ mới, dịch vụ mới được gọi backend nào, được truy cập CSDL/Redis/Kafka/MinIO nào, và có cần luồng gửi ra ngoài hay không.
    \item Triển khai và kiểm tra. Sau khi pod chạy, cần xác nhận Service có endpoint, Vault cấp được bí mật, SPIRE cấp được SVID, Cilium không chặn nhầm luồng hợp lệ, Trivy sinh báo cáo khi quét xong và PDP gán nhãn tin cậy cho pod.
\end{enumerate}

\section{Cập nhật frontend trên Cloudflare Pages}
Hai frontend \texttt{atd\_frontend} và \texttt{rct\_frontend} không chạy trong cụm Kubernetes. Khi thay đổi giao diện hoặc nối thêm API mới, cần cập nhật mã nguồn frontend, biến môi trường public và triển khai lại qua Cloudflare Pages:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
$ cd atd_frontend
$ npm run pages:build
$ npm run pages:deploy

$ cd ../rct_frontend
$ npm run pages:build
$ npm run pages:deploy
\end{minted}

Các biến như \texttt{NEXT\_PUBLIC\_API\_BASE\_URL}, \texttt{NEXT\_PUBLIC\_USE\_MOCK} và cấu hình Keycloak cần được đặt đúng trước khi build vì chúng được đưa vào gói frontend tại thời điểm đóng gói. Sau khi frontend được triển khai, trình duyệt tải giao diện từ Cloudflare Pages, gọi Keycloak để đăng nhập và gửi yêu cầu nghiệp vụ tới Kong Gateway. Vì vậy, nếu thêm API mới cho frontend, phải kiểm tra đồng thời route ở Kong, policy OPA, quyền Keycloak và chính sách Cilium phía backend.

\section{Giám sát luồng mạng qua giao diện trực quan}
Để phân tích luồng dữ liệu hoặc xử lý sự cố kết nối giữa các microservices, quản trị viên sử dụng giao diện đồ họa của Hubble theo các bước sau:
\begin{enumerate}
    \item Kích hoạt chuyển tiếp cổng (port-forward) từ máy trạm đến dịch vụ Hubble UI trong cụm:
\begin{minted}[fontsize=\scriptsize,breaklines]{bash}
$ KUBECONFIG=/home/ptb/.kube/config-job7189 \
  kubectl port-forward -n kube-system svc/hubble-ui 12000:80
\end{minted}
    \item Mở trình duyệt web và truy cập địa chỉ \texttt{http://localhost:12000}.
    \item Chọn không gian tên \texttt{job7189-apps} ở góc trên bên trái để quan sát sơ đồ kết nối thời gian thực giữa các microservices nghiệp vụ. Các kết nối bị chặn bởi chính sách Cilium sẽ hiển thị màu đỏ (dropped), các luồng hợp lệ sẽ hiển thị màu xanh (forwarded), giúp dễ dàng định vị lỗi cấu hình chính sách.
\end{enumerate}
