\section{Lộ trình triển khai Zero Trust theo NIST SP 800-207}

NIST SP 800-207 (Mục 7 — ``Migrating to a Zero Trust Architecture'') khuyến nghị doanh nghiệp triển khai ZTA theo một lộ trình tuần tự, không phải ``bật tắt một công tắc duy nhất'' \cite{nist800207}. Hệ thống JOB7189 tuân thủ lộ trình 6 giai đoạn sau:

\begin{table}[H]
\centering
\caption{Lộ trình triển khai ZTA cho JOB7189 theo NIST SP 800-207 Mục 7}
\label{tab:nist_roadmap}
\begin{tabularx}{\textwidth}{|c|p{3.8cm}|X|p{2.5cm}|}
\hline
\textbf{Bước} & \textbf{Giai đoạn NIST} & \textbf{Hiện thực hóa trong JOB7189} & \textbf{Script / File} \\

1 & Kiểm kê tài sản và Xác định luồng giao dịch & Liệt kê 7 microservice, 5 namespace, vẽ sơ đồ N-S/E-W & Mục 3.2 \\
\hline

\hline
3 & Thiết kế và áp dụng chính sách truy cập & Kong JWT (N-S), Cilium Default Deny + Allow-Explicit (E-W) & \texttt{kong.yml}, \texttt{20-security-policies} \\

\hline
\end{tabularx}
\end{table}

Lộ trình này đảm bảo rằng mỗi giai đoạn là \textbf{tiền đề} cho giai đoạn tiếp theo: không thể thiết lập microsegmentation (Bước 4) nếu chưa có gốc tin cậy (Bước 2); không thể xác minh (Bước 6) nếu chưa có observability (Bước 5).

\subsection{Sơ đồ luồng giao dịch: North-South và East-West}

\textbf{Luồng North-South} (Client → Hệ thống):
\begin{enumerate}
    \item Client (trình duyệt, Postman, mobile app) gửi HTTPS request.
    \item Nginx Ingress Controller (\texttt{ingress-nginx}) tiếp nhận, route theo domain:
    \begin{itemize}
        \item \texttt{api.job7189.com} → Kong Gateway (\texttt{namespace: gateway})
        \item \texttt{auth.job7189.local} → Keycloak (\texttt{namespace: security})
    \end{itemize}
    \item Kong Gateway xác thực JWT (nếu route yêu cầu) rồi proxy xuống backend service.
    \item Backend service xử lý logic nghiệp vụ, truy xuất database qua Vault dynamic credentials.
\end{enumerate}

\textbf{Luồng East-West} (Service → Service nội bộ):
\begin{itemize}
    \item \texttt{job-service} → \texttt{GET /api/v1/internal/workspaces/\{id\}} → \texttt{workspace-service}
    \item \texttt{hiring-service} → \texttt{GET /api/v1/internal/profile/\{id\}} → \texttt{identity-service}
    \item Mọi service → \texttt{TCP 3306} → MySQL (\texttt{namespace: data})
    \item Mọi service → \texttt{TCP 8200} → Vault (\texttt{namespace: vault})
    \item Mọi service → \texttt{TCP 9092} → Kafka (\texttt{namespace: data})
\end{itemize}

\section{Bước 2: Thiết lập gốc tin cậy — Identity Layer}

Theo NIST SP 800-207, gốc tin cậy (Root of Trust) phải được thiết lập \textbf{trước mọi thành phần khác}. Trong JOB7189, gốc tin cậy gồm 3 thành phần: Keycloak (User Identity), Vault (Workload Secrets), và Kubernetes ServiceAccount (Workload Identity).

\subsection{Keycloak: User Identity qua OIDC/JWT}

Keycloak được triển khai tại namespace \texttt{security} dưới dạng custom image (\texttt{job7189/keycloak-custom:v1.0}) với realm \texttt{job7189} được import tự động trong quá trình bootstrap.

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# infras/k8s-yaml/02-keycloak.yaml (trích)
containers:
- name: keycloak
  image: 172.17.0.1:5000/job7189/keycloak-custom:v1.0
  command: ["/opt/keycloak/bin/kc.sh", "start-dev", "--import-realm"]
  env:
  - name: KC_DB_URL
    value: "jdbc:mysql://mysql.data.svc.cluster.local:3306/job7189keycloak"
  - name: KC_DB_PASSWORD              # ZTA: inject từ K8s Secret
    valueFrom:
      secretKeyRef:
        name: app-secrets
        key: mysql-root-password
  - name: KEYCLOAK_ADMIN_PASSWORD     # ZTA: inject từ K8s Secret
    valueFrom:
      secretKeyRef:
        name: app-secrets
        key: keycloak-admin-password
\end{minted}
\caption{Keycloak Deployment — mật khẩu được inject từ Kubernetes Secret, không hardcode}
\label{lst:keycloak_deploy}
\end{listing}

\textbf{Đặc điểm ZTA:} Mật khẩu quản trị và database đều được sinh ngẫu nhiên tại thời điểm deploy:

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{bash}
# 02-deploy-infrastructure.sh — Sinh credentials ngẫu nhiên
KEYCLOAK_ADMIN_PASS=$(openssl rand -base64 16 | tr -d '/+=' | head -c 16)
MYSQL_ROOT_PASS=$(openssl rand -base64 24 | tr -d '/+=' | head -c 24)
kubectl create secret generic app-secrets --namespace=security \
  --from-literal=mysql-root-password="$MYSQL_ROOT_PASS" \
  --from-literal=keycloak-admin-password="$KEYCLOAK_ADMIN_PASS" \
  --dry-run=client -o yaml | kubectl apply -f -
\end{minted}
\caption{Sinh mật khẩu ngẫu nhiên \texttt{openssl rand} — không có mật khẩu hardcode trong mã nguồn}
\label{lst:credential_gen}
\end{listing}

\textbf{Quy trình xác thực OIDC/JWT:}
\begin{enumerate}
    \item Client gọi Keycloak endpoint \texttt{/realms/job7189/protocol/openid-connect/token} với OAuth 2.0 flow.
    \item Keycloak trả về Access Token (JWT) chứa: \texttt{sub}, \texttt{iss}, \texttt{realm\_access.roles}, \texttt{exp}.
    \item Client gửi JWT trong header \texttt{Authorization: Bearer <token>}.
    \item Kong Gateway xác minh chữ ký RS256 bằng Public Key từ Keycloak JWKS endpoint.
\end{enumerate}

\subsection{HashiCorp Vault: Workload Identity và Dynamic Secrets}

Vault được triển khai theo mô hình \textbf{Dual-Vault Architecture}: Vault Dev (Transit Auto-Unseal engine) + Vault Production (TLS, PVC 2Gi):

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{text}
Kiến trúc Vault:

  vault-dev (Dev mode, port 8300)
    └── Transit Engine: key "vault-prod-unseal-key"
        └── Auto-unseal cho vault-prod

  vault-prod (StatefulSet, port 8200 TLS)
    ├── Storage: file backend trên PVC 2Gi
    ├── TLS: cert-manager self-signed certificate
    ├── Auth: Kubernetes Auth Method
    ├── Secret Engines:
    │   ├── KV v2 (path: secret/) — APP_KEY, Keycloak secrets
    │   └── Database (MySQL dynamic credentials)
    └── Policies: 7 service policies + keycloak + mysql-bootstrap
\end{minted}
\caption{Kiến trúc Dual-Vault: Transit Auto-Unseal + Production Mode TLS}
\label{lst:vault_arch}
\end{listing}

\textbf{Kubernetes Auth Method:} Vault xác thực workload dựa trên Kubernetes ServiceAccount — mỗi service có đúng 1 ServiceAccount, được map với đúng 1 Vault role, chỉ có quyền đọc đúng 1 database credential:

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{bash}
# 99-fast-rebuild-vault.sh — Tạo policy và role cho mỗi service
for SVC in identity-service workspace-service job-service \
           hiring-service candidate-service \
           communication-service storage-service; do
  vault policy write "$SVC" - <<POL
path "secret/data/laravel-common" { capabilities = ["read"] }
path "database/creds/${SVC}" { capabilities = ["read"] }
path "secret/data/${SVC}" { capabilities = ["read"] }
POL
  vault write auth/kubernetes/role/"$SVC" \
    bound_service_account_names="$SVC" \
    bound_service_account_namespaces=job7189-apps \
    policies="$SVC" \
    ttl=1h max_ttl=24h
done
\end{minted}
\caption{Vault RBAC: mỗi service có policy riêng — \texttt{identity-service} không thể đọc \texttt{database/creds/hiring-service}}
\label{lst:vault_rbac}
\end{listing}

\textbf{Dynamic Database Credentials — Just-In-Time (JIT):}

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{bash}
# Cấu hình Database Engine — 7 role, mỗi role gắn đúng 1 database
vault write database/config/mysql \
  plugin_name=mysql-database-plugin \
  connection_url='{{username}}:{{password}}@tcp(mysql.data.svc:3306)/' \
  allowed_roles='*' username='vault_manager' password='${VAULT_MGR_PASS}'

# Ví dụ: role identity-service — TTL 1h, chỉ 4 quyền CRUD trên 1 DB
vault write database/roles/identity-service \
  db_name=mysql \
  username_template='usr_{{random 16 | truncate 12}}' \
  creation_statements="CREATE USER '{{name}}'@'%' IDENTIFIED BY '{{password}}';
    GRANT SELECT,INSERT,UPDATE,DELETE ON job7189_identity_db.* TO '{{name}}'@'%';" \
  revocation_statements="DROP USER IF EXISTS '{{name}}'@'%';" \
  default_ttl='1h' max_ttl='24h'
\end{minted}
\caption{Vault Database Engine: mỗi service nhận MySQL user ngẫu nhiên, TTL 1h, chỉ 4 quyền CRUD}
\label{lst:vault_db_engine}
\end{listing}

\textbf{Vault Agent Injector — Sidecar pattern cho secret injection:}

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# k8s-management/charts/laravel-app/templates/deployment.yaml (trích)
annotations:
  vault.hashicorp.com/agent-inject: "true"
  vault.hashicorp.com/agent-init-first: "true"
  vault.hashicorp.com/role: "{{ .Release.Name }}"
  vault.hashicorp.com/tls-skip-verify: "true"
  vault.hashicorp.com/agent-inject-status: "update"   # Cho phép rotation
  # Dynamic DB credentials ghi vào tmpfs
  vault.hashicorp.com/agent-inject-secret-db: "database/creds/{{ .Release.Name }}"
  vault.hashicorp.com/agent-inject-file-db: ".env.db"
  vault.hashicorp.com/agent-inject-template-db: |-
    {{- with secret "database/creds/<service-name>" -}}
    DB_USERNAME="{{ .Data.username }}"
    DB_PASSWORD="{{ .Data.password }}"
    LEASE_ID="{{ .LeaseID }}"
    {{ end }}
\end{minted}
\caption{Vault Agent Injector annotations — Dynamic DB credentials trên tmpfs}
\label{lst:vault_annotations}
\end{listing}

\textbf{Quy trình vòng đời JIT credential} (xem Hình~\ref{fig:vault_jit_lifecycle}):
\begin{enumerate}
    \item Pod được tạo → Vault Mutating Webhook inject init container \texttt{vault-agent-init}.
    \item Init container xác thực với Vault qua Kubernetes ServiceAccount token (\texttt{auth/kubernetes}).
    \item Vault tạo MySQL user mới (\texttt{usr\_<random12>}), cấp 4 quyền CRUD trên đúng 1 database → ghi vào \texttt{/vault/secrets/.env.db} trên \textbf{tmpfs} (emptyDir medium: Memory).
    \item Sidecar \texttt{env-loader} giám sát MD5 checksum, merge \texttt{.env.common} + \texttt{.env.db} + \texttt{.env.extra} → \texttt{/app-secrets/.env}.
    \item Application container đọc \texttt{/app-secrets/.env}, kết nối MySQL với credential động.
    \item Sidecar \texttt{env-watcher} phát hiện lease rotation → gọi \texttt{/api/internal/reload-db} → Laravel reconnect.
    \item TTL hết hạn (1h) → Vault revoke credential cũ (\texttt{DROP USER IF EXISTS}) → Agent tự động lấy credential mới.
\end{enumerate}

\subsection{Kong Gateway: PEP cho North-South — JWT validation và route-level enforcement}

Kong Gateway 3.6 được triển khai ở chế độ DB-less (Declarative Mode), toàn bộ cấu hình trong file \texttt{kong.yml} mount qua ConfigMap:

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# infras/kong/kong.yml (trích)
consumers:
  - username: keycloak-user
    custom_id: keycloak-issuer

\section{Bước 4: Triển khai Microsegmentation — Cilium Network Policy}

Đây là phần \textbf{cốt lõi} của kiến trúc Zero Trust: chuyển từ ``Flat Network'' (mạng phẳng, mọi Pod tự do giao tiếp) sang ``Microsegmented Network'' (mỗi Pod bị cô lập, chỉ giao tiếp qua các kênh được khai báo tường minh). Hệ thống sử dụng \textbf{CiliumNetworkPolicy} với công nghệ eBPF hoạt động tại tầng kernel \cite{cilium_docs}.

\subsection{Quy trình triển khai microsegmentation}

Microsegmentation được triển khai qua script \texttt{apply-zta-microsegmentation.sh} theo thứ tự nghiêm ngặt:

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{bash}
#!/bin/bash
# infras/k8s-yaml/cilium-policies/apply-zta-microsegmentation.sh
echo "🛡️ ZTA Microsegmentation đang được kích hoạt..."
kubectl apply -f 00-default-deny.yaml          # Bước 1: Khóa toàn bộ
kubectl apply -f 01-allow-egress-dns.yaml       # Bước 2: Mở DNS
kubectl apply -f 02-allow-egress-data.yaml      # Bước 3: Mở data/vault
kubectl apply -f 03-allow-ingress-kong.yaml     # Bước 4: Mở gateway ingress
kubectl apply -f 04-allow-internal-api-strict.yaml  # Bước 5: Mở E-W có kiểm soát
echo "✅ Microsegmentation cho namespace 'job7189-apps' đã được bật!"
\end{minted}
\caption{Script triển khai microsegmentation — thứ tự: khóa trước, mở sau}
\label{lst:apply_microseg}
\end{listing}

\textbf{Thứ tự này cực kỳ quan trọng:} Default Deny phải được áp dụng \textbf{trước tiên} (Listing~\ref{lst:default_deny_actual}). Nếu áp dụng Allow-rules trước mà chưa có Default Deny, traffic vẫn đi qua bình thường và microsegmentation vô nghĩa.


\subsection{Lớp 1: Default Deny — Khóa toàn bộ mạng}

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}

  name: "default-deny-all"
  namespace: "job7189-apps"

    matchLabels: {}       # Áp dụng cho TẤT CẢ pods
  ingress: []             # Chặn toàn bộ Ingress
  egress: []              # Chặn toàn bộ Egress
\end{minted}
\caption{CiliumNetworkPolicy Default Deny — chặn 100\% traffic mặc định}

\subsection{Lớp 5: ServiceAccount-based micro-segmentation (East-West chi tiết)}

Ngoài Lớp 1-4 áp dụng theo \textit{namespace}, hệ thống thiết kế thêm lớp micro-segmentation theo \textbf{ServiceAccount identity} — xác định chính xác \textit{service nào} được nói chuyện với \textit{service nào}:

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# 20-security-policies.yaml — Chỉ identity-service → MySQL (port 3306)
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-identity-to-mysql
  namespace: job7189-apps
spec:
  endpointSelector:
    matchLabels:
      io.cilium.k8s.policy.serviceaccount: identity-service
  egress:
  - toEndpoints:
    - matchLabels:
        io.cilium.k8s.policy.serviceaccount: mysql
        io.kubernetes.pod.namespace: data
    toPorts:
    - ports:
      - port: "3306"
        protocol: TCP
---
# Phía MySQL cũng chỉ chấp nhận từ identity-service
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: allow-mysql-from-identity
  namespace: data
spec:
  endpointSelector:
    matchLabels:
      io.cilium.k8s.policy.serviceaccount: mysql
  ingress:
  - fromEndpoints:
    - matchLabels:
        io.cilium.k8s.policy.serviceaccount: identity-service
        io.kubernetes.pod.namespace: job7189-apps
    toPorts:
    - ports:
      - port: "3306"
        protocol: TCP
\end{minted}
\caption{Micro-segmentation theo ServiceAccount: policy kép (egress + ingress) cho cặp identity→MySQL}
\label{lst:sa_microseg}
\end{listing}

\textbf{Quan trọng:} Hệ thống sử dụng \textbf{policy kép} — cả nguồn (egress) lẫn đích (ingress) đều phải có rule. Nếu \texttt{job-service} cố kết nối đến MySQL mà không có policy cho egress của nó, gói tin bị Drop ở phía gửi. Nếu có egress nhưng MySQL không có ingress rule cho \texttt{job-service}, gói tin bị Drop ở phía nhận.


\subsection{Lớp 6: L7 HTTP Policy — kiểm soát method và path}

Không chỉ kiểm soát ``ai nói chuyện với ai'', Cilium L7 còn kiểm soát ``nói cái gì'':

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# 04-allow-internal-api-strict.yaml
# job-service chỉ được gọi GET /api/v1/internal/workspaces/* → workspace-service
apiVersion: "cilium.io/v2"
kind: CiliumNetworkPolicy
metadata:
  name: "allow-internal-job-to-workspace"
  namespace: "job7189-apps"
spec:
  endpointSelector:
    matchLabels:
      app: workspace-service          # Đích
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: job-service              # Nguồn: CHỈ job-service
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      rules:
        http:
        - method: "GET"               # CHỈ GET
          path: "/api/v1/internal/workspaces/.*"   # CHỈ path này
---
# hiring-service chỉ GET /api/v1/internal/profile/* → identity-service
apiVersion: "cilium.io/v2"
kind: CiliumNetworkPolicy
metadata:
  name: "allow-internal-identity"
  namespace: "job7189-apps"
spec:
  endpointSelector:
    matchLabels:
      app: identity-service
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: hiring-service
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/api/v1/internal/profile/.*"
\end{minted}
\caption{L7 HTTP Policy: kiểm soát cả HTTP method lẫn API path cho luồng East-West}
\label{lst:l7_policy}
\end{listing}

\textbf{Kịch bản tấn công bị chặn:} Nếu \texttt{job-service} bị xâm nhập:
\begin{itemize}
    \item Gửi \texttt{POST /api/v1/internal/workspaces/1/delete} → \textbf{403 Denied} (chỉ GET được phép)
    \item Gửi \texttt{GET /api/admin/users} → \textbf{403 Denied} (path không khớp pattern)
    \item Gửi request đến \texttt{candidate-service} → \textbf{Drop} (không có policy cho cặp này)
\end{itemize}

\begin{listing}[H]
\begin{minted}[fontsize=\footnotesize]{yaml}
# L7 restriction trên identity-service (20-security-policies.yaml)
spec:
  endpointSelector:
    matchLabels:
      io.cilium.k8s.policy.serviceaccount: identity-service
  ingress:
  - fromEndpoints:
    - matchExpressions:
      - key: io.cilium.k8s.policy.serviceaccount
        operator: Exists          # Mọi SA đã xác thực
    toPorts:
    - ports:
      - port: "80"
      rules:
        http:
        - method: "GET"
          path: "/api/v1/auth"    # CHỈ endpoint này
\end{minted}
\caption{L7 restriction: identity-service chỉ expose \texttt{GET /api/v1/auth} cho các service nội bộ}
\label{lst:l7_identity}
\end{listing}
