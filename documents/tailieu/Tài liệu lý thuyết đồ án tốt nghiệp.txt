* Network Policy trong Kubernetes: mô hình kiểm soát truy cập tầng mạng
* Centralized Logging và Security Observability: nguyên lý thu thập và phân tích sự kiện bảo mật tập trung

3.1. Ánh xạ NIST SP 800-207 vào stack công nghệ
* Policy Engine (PE): vai trò của Keycloak và Cilium
* Policy Administrator (PA): cơ chế đồng bộ và phân phối policy
* Policy Enforcement Point (PEP): Cilium tại tầng mạng, Tetragon tại tầng runtime
3.2. Môi trường thực nghiệm
* Cấu hình Kubernetes multi-node cluster
* Phân bổ tài nguyên cho các thành phần
* Kiến trúc triển khai tổng thể
3.3. Thiết kế và triển khai kiểm soát truy cập tầng mạng
* Nguyên tắc Deny-All / Allow-Explicit
* Thiết kế Network Policy L3/L4 cho từng cặp service
* Thiết kế Network Policy L7 cho các endpoint nhạy cảm
* Xử lý đặc thù của Message Broker traffic
* Kiểm tra: đúng endpoint đi qua, sai endpoint bị block
3.4. Thiết kế và triển khai xác thực danh tính liên dịch vụ
* Thiết kế mTLS Sidecarless: cấp phát, phân phối và rotation certificate
* Tích hợp JWT từ Identity Provider vào kiểm soát truy cập L7
* Verify xác thực hai chiều giữa các service
3.5. Thiết kế và triển khai Runtime Security
* Các hành vi cần giám sát: syscall nguy hiểm, network call bất thường, truy cập file nhạy cảm
* Thiết kế và cấu hình TracingPolicy
* Pipeline phát hiện và phản hồi: Tetragon → ELK → Alert
3.6. Thiết kế và triển khai Observability
* Kiến trúc tổng thể: phân tách Observer khỏi hệ thống được giám sát
* Centralized Logging: thu thập và truy vấn security events
* Security metrics: Grafana dashboard cho Cilium và Tetragon
* Pipeline cảnh báo tự động qua email

Phân tích các phương án tích hợp HashiCorp Vault trong Kubernetes
1. Tổng quan bài toán quản lý secret
Trong kiến trúc microservices triển khai trên Kubernetes, hệ thống cần quản lý hai nhóm bí mật chính:
* Static secrets: các thông tin cố định, ít thay đổi
   * OAuth2 client secret (oauth2-proxy)
   * Cookie secret
   * Mật khẩu cơ sở dữ liệu (Keycloak, MySQL root)
* Dynamic secrets: thông tin được sinh động theo phiên làm việc
   * Tài khoản MySQL tạm thời cho từng microservice
   * Tự động hết hạn (TTL) và thu hồi
Cơ chế này phù hợp với mô hình Zero Trust Architecture (ZTA), trong đó mỗi service chỉ được cấp quyền tối thiểu trong một khoảng thời gian hữu hạn Zero Trust Architecture.

Luồng hoạt động Vault với Laravel
________________


Giải thích kỹ
Có 4 thành phần tham gia:
Vault Server     — lưu trữ và cấp phát secrets
Vault Agent      — sidecar chạy cùng pod với Laravel
K8s API Server   — xác thực identity của pod
Laravel          — không biết Vault tồn tại


Bước 1 — Pod start, Vault Agent khởi động trước
Kubernetes inject Vault Agent vào pod như một init container và sidecar container. Vault Agent chạy trước Laravel.
Bước 2 — Vault Agent authenticate với Vault Server
Vault Agent đọc K8s ServiceAccount token
    │ token này được K8s tự động mount vào mọi pod
    │ tại /var/run/secrets/kubernetes.io/serviceaccount/token
    ▼
Gửi token lên Vault Server: POST /auth/kubernetes/login
    │
    ▼
Vault Server gọi K8s API: "Token này có hợp lệ không? Pod này thuộc ServiceAccount nào?"
    │
    ▼
K8s API xác nhận → Vault kiểm tra policy
"ServiceAccount identity-service ở namespace job7189-app được phép đọc database/creds/identity-service"
    │
    ▼
Vault trả về Vault Token cho Agent


Bước 3 — Vault Agent lấy dynamic credentials từ Database Engine
Vault Agent dùng Vault Token gọi:
GET /database/creds/identity-service
    │
    ▼
Vault Database Engine kết nối MySQL
    └── CREATE USER 'v-identity-abc123'@'%' IDENTIFIED BY 'xyz789'
    └── GRANT SELECT,INSERT,UPDATE,DELETE ON identity_db.* TO ...
    │
    ▼
Vault trả về:
  username: v-identity-abc123
  password: xyz789
  lease_id: database/creds/identity-service/abc...
  lease_duration: 1h


Bước 4 — Vault Agent render vào file .env
Vault Agent dùng template engine để format output:
Template định nghĩa trong annotation:
{{ with secret "database/creds/identity-service" }}
DB_CONNECTION=mysql
DB_HOST=mysql.data.svc.cluster.local
DB_PORT=3306
DB_DATABASE=identity_db
DB_USERNAME={{ .Data.username }}
DB_PASSWORD={{ .Data.password }}
{{ end }}


Render ra file /var/www/html/.env.vault — mount vào đúng thư mục Laravel.
Bước 5 — Laravel start và đọc file
Container Laravel start sau khi Vault Agent init container hoàn thành. Laravel đọc .env như bình thường — không biết credentials đến từ Vault.
Bước 6 — Vault Agent tự renew
Sau 40 phút (2/3 của 1h lease)
    │
    ▼
Vault Agent tự gọi: PUT /sys/leases/renew
    │
    ▼
Vault gia hạn credential thêm 1h
    └── MySQL user vẫn còn
    └── Laravel không bị disconnect
    └── Không cần restart pod


Bước 7 — Khi lease hết hạn hoàn toàn (max_ttl = 24h)
Vault revoke credential
    └── MySQL user bị xóa
    └── Vault Agent lấy credential mới
    └── Render lại file .env.vault
    └── Laravel cần reload — entrypoint script handle việc này

Đoạn tóm tắt cho báo cáo
Trong hệ thống, HashiCorp Vault được tích hợp với Laravel microservices thông qua cơ chế Vault Agent Sidecar. Khi một pod khởi động, Vault Agent — chạy như một init container — tự động xác thực danh tính với Vault Server bằng Kubernetes ServiceAccount token, một cơ chế xác thực built-in của Kubernetes không yêu cầu bất kỳ credential tĩnh nào. Sau khi xác thực thành công, Vault Agent yêu cầu Database Secrets Engine tạo một MySQL user tạm thời với thời hạn sử dụng giới hạn (TTL), sau đó render thông tin này vào file .env của Laravel theo định dạng chuẩn. Laravel đọc file này khi khởi động và kết nối database như bình thường, hoàn toàn không biết về sự tồn tại của Vault. Vault Agent duy trì hoạt động như một sidecar container, tự động gia hạn credential trước khi hết hạn để đảm bảo tính liên tục của dịch vụ. Khi credential đạt max TTL, Vault thu hồi MySQL user cũ, tạo user mới và cập nhật lại file cấu hình. Cơ chế này đảm bảo không có credential tĩnh nào tồn tại trong hệ thống, phù hợp với nguyên tắc Zero Trust Architecture về việc tối thiểu hóa thời gian tồn tại của secrets.

6.1 Triển Khai Blue-Green (Xanh-Lục)
Chiến lược Blue-Green yêu cầu duy trì hai hệ thống môi trường song song và giống hệt nhau về mặt cơ sở hạ tầng, được gọi tên là "Blue" và "Green". Trong điều kiện vận hành bình thường, toàn bộ lưu lượng truy cập thực tế của người dùng được định tuyến thẳng vào môi trường Blue (hiện đang chứa phiên bản phần mềm ổn định). Khi đội ngũ phát triển tung ra bản nâng cấp mới, mã nguồn này sẽ được triển khai vào môi trường Green hoàn toàn nhàn rỗi. Môi trường Green lúc này được bảo vệ khỏi lưu lượng công cộng, cho phép đội ngũ thực thi các bài kiểm tra toàn diện, mô phỏng tải trọng thực tế.
Điểm kỳ diệu của chiến lược này nằm ở thời điểm chuyển giao (cutover). Khi chất lượng của môi trường Green được xác nhận, các kỹ sư tiến hành điều chỉnh cấu hình tại tầng định tuyến mạng (thông qua cập nhật bản ghi DNS hoặc thay đổi đích đến của bộ cân bằng tải - Load Balancer). Lưu lượng truy cập chuyển dịch lập tức 100% từ Blue sang Green, biến Green trở thành môi trường phục vụ sản xuất mới mà không làm gián đoạn bất kỳ kết nối nào của người dùng (Zero-Downtime). Môi trường Blue cũ lúc này chuyển sang trạng thái chờ. Lợi thế bảo hiểm tuyệt đối của Blue-Green là khả năng Rollback tức thời: nếu phiên bản mới tại Green phát sinh sự cố bất ngờ, kỹ sư chỉ cần một thao tác đổi cấu hình mạng để chuyển lại lưu lượng về môi trường Blue ổn định cũ ngay lập tức, bỏ qua mọi tiến trình biên dịch hay cài đặt lại phức tạp.
Trong hệ sinh thái Kubernetes, cấu trúc Blue-Green được thao tác rất tinh gọn thông qua định tuyến bộ chọn (Label Selectors) của Service hoặc thông qua thiết lập tệp Ingress YAML. Kỹ sư chỉnh sửa tham số service: name: trong cấu hình Ingress từ myapp-blue sang myapp-green, kích hoạt sự chuyển đổi lưu lượng ở cấp độ cụm (cluster-level) chỉ trong tích tắc
Argo Rollouts
7.1 Lộ Trình Triển Khai Mở Rộng Và Thu Hẹp (Expand and Contract Pattern)
Kỹ thuật này phá vỡ một hành động di trạch phá hủy (destructive migration, ví dụ đổi tên một cột) thành nhiều chu kỳ phát hành an toàn độc lập.
   * Giai đoạn Mở Rộng (Expand): Triển khai một tệp kịch bản Migration để tạo ra một cột dữ liệu mới song song với cột cũ. Cột mới này phải được cấu hình linh hoạt cho phép giá trị Null. Các phiên bản ứng dụng Blue hiện hành hoàn toàn không nhận thức được sự tồn tại của cột này và tiếp tục hoạt động mà không sinh lỗi.
   * Giai đoạn Dịch Chuyển (Deploy Green): Phát hành bản nâng cấp ứng dụng mới (Green). Tại bước này, cấu trúc mã nguồn được chỉnh sửa để áp dụng mô hình "Ghi Kép" (Dual Writes) - ứng dụng sẽ đồng thời ghi dữ liệu cập nhật vào cả cột cũ và cột mới, đồng thời thực thi kịch bản chuyển đổi dần dữ liệu lịch sử từ cột cũ sang cột mới.
   * Giai đoạn Chuyển Đổi (Switch Traffic): Cấu hình lại bộ định tuyến để đưa toàn bộ lưu lượng công cộng sang hệ thống Green.
   * Giai đoạn Thu Hẹp (Contract): Trong chu kỳ phát hành tiếp theo, sau khi hệ thống đã loại bỏ hoàn toàn mã nguồn bản Blue cũ và ứng dụng Green chứng minh độ tin cậy khi giao tiếp với cột dữ liệu mới, các kỹ sư mới khởi chạy kịch bản Migration cuối cùng nhằm xóa bỏ tận gốc cột dữ liệu cũ, hoàn tất quy trình. Quá trình này được hỗ trợ mạnh mẽ bởi các bộ công cụ di trạch trực tuyến như pt-online-schema-change hoặc thiết lập cô lập quy trình thông qua các ECS Task chạy một lần nhằm kiểm soát tiến trình db:migrate trước khi mã nguồn ứng dụng thực thi.
